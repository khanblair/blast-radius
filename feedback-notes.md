# Feedback Notes

Running log of friction encountered while building against DataHub Core + the DataHub MCP Server, for the Feedback Prize submission and potential upstream OSS contribution.

The general, DataHub-specific *technique* this friction fed into — resolving entities, scoping column-level lineage, structurally distinguishing origin vs. cascade breaks, weighting by usage/exposure/ownership — is packaged separately as a reusable skill at [`skills/datahub-impact-analysis/`](skills/datahub-impact-analysis/SKILL.md), independent of this project's own dbt/Postgres demo estate.

## 2026-07-07 — Phase 1 (Week 1 Foundation)

- **No PAT needed for local quickstart writes.** `datahub docker quickstart` ships with Metadata Service Auth off by default, so both `datahub ingest` and direct `DatahubRestEmitter` writes to GMS succeed with no token at all. The frontend GraphQL layer enforces its own separate policy check (see next item), which is easy to conflate with GMS-level auth.
- **The seeded root `datahub` user does not have `generatePersonalAccessTokens` by default** in v1.5.0.6 (`platformPrivileges.generatePersonalAccessTokens` is `false` even when logged in as `datahub`/`datahub`). Minting a PAT via the `createAccessToken` GraphQL mutation with that identity returns a 403 `UNAUTHORIZED`. Worth documenting clearly in DataHub's own quickstart docs, since the "generate via UI" path implies this identity has admin rights, but token self-issuance is gated separately.
- **Postgres + dbt ingestion do not sibling-merge into one URN by default.** Even with `env`, `convert_urns_to_lowercase`, and `target_platform` aligned across both recipes exactly as documented, no `Siblings` aspect was created. Instead, DataHub creates two parallel entities per table — `urn:li:dataset:(urn:li:dataPlatform:postgres,...)` (physical) and `urn:li:dataPlatform:dbt,...)` (logical, carries the real `ref()`/`source()` dependency graph) — bridged by a single `upstreamLineage` edge from the postgres entity to its dbt counterpart. The dbt-platform entity's own `upstreamLineage` then points at the *target-platform* (postgres) URNs of its actual dependencies, not at sibling dbt-platform URNs. Full lineage traversal from a raw table to a downstream mart is still possible, but the path alternates platforms hop-by-hop — worth knowing before writing lineage-traversal logic (Assessment Engine, Loop 2) so hop-depth budgeting accounts for these bridge hops.
- **`DataJob.inlets`/`.outlets` (the `datahub.api.entities.datajob` convenience helper) silently accept plain URN strings at assignment time but throw `AttributeError: 'str' object has no attribute 'to_key_aspect'` at `.emit()`** — they actually require `DatasetUrn` objects (`DatasetUrn.from_string(...)`). The type hints say this (`List[DatasetUrn]`), but nothing rejects a string until the emit call three steps later, so the failure surfaces far from its cause.
- **`DataFlow(..., cluster="X", env="X")` raises `ValueError: Cannot provide both env and cluster parameter`** — `cluster` is deprecated in favor of `env` as of the installed SDK version (1.6.0.10), but an example combining both is easy to write by analogy with older docs/blog posts still floating around.
- **Newly `claude mcp add`-ed servers aren't visible to the current session's tool search** — they connect fine (`claude mcp list` shows ✔ Connected) but the running conversation can't call their tools until a fresh session starts. Had to verify the DataHub MCP server's actual `tools/call` path with a standalone `mcp` Python SDK stdio client instead of relying on `claude mcp list`'s "Connected" status as proof the tools work end-to-end.

## 2026-07-08 — Phases 2-8 (Assessment through Watch Mode)

- **Different `mcp-server-datahub` tools shape their result entities inconsistently.** `get_lineage`'s result entities carry a `"type"` field (`"DATASET"`, `"DASHBOARD"`, ...); `search`'s result entities do not — the same logical "entity" object has a different shape depending on which tool returned it. Code written against one tool's shape (`entity.get("type") == "DATASET"`) silently misbehaves when reused against another's results (it doesn't error, it just never matches, so URN resolution quietly returned nothing until a live run caught it). Worth documenting per-tool result shapes explicitly rather than implying a single uniform "entity" schema across the tool surface.
- **`list_schema_fields` (the tool needed for schema-diffing/watch-mode use cases) isn't obviously discoverable from tool descriptions alone** — confirming its existence and exact argument shape required reading `mcp-server-datahub`'s own source in the local `uv` package cache rather than a docs page or the tool list's descriptions. A short "which tool do I want for X" table (schema fields vs. lineage vs. usage vs. ownership) in the server's own README would have saved this detour.
- **Column-level lineage scoping and dataset-level ("broad") reachability from the *same* root URN don't always agree with what the dbt-level `ref()`/`source()` dependency graph alone would predict.** While building this project's own worked examples (`examples/`), a declared change on `raw_payments.payment_method` correctly showed `stg_payments` as a direct hard break, but *also* showed `fct_revenue` (a mart that, per the dbt project's own `ref()` graph, only depends on `fct_orders` and `dim_customers` — neither of which depends on `raw_payments` or `stg_payments`) as an affected, `SELECT *`-exposed asset. This is plausibly explained by this estate's own synthetic Airflow pipeline metadata (`estate/metadata/emit_pipeline.py`'s `DataFlow`/`DataJob`, emitted in Phase 1) creating job-level lineage edges broader than the real per-model dbt dependency graph — but this wasn't conclusively root-caused before submission. Flagging as an open question for anyone else composing dbt-level and pipeline-level (Airflow/Dagster) lineage into the same DataHub graph: know which lineage *source* (dbt-emitted vs. pipeline-emitted) is answering a given downstream-reachability query, since they can disagree.
- **`dbt-core` treats the `DBT_PROJECT_DIR` environment variable as a real override for project-directory resolution, and it wins over plain `cwd`-based detection when no `--project-dir` flag is passed.** Not a DataHub issue, but relevant to anyone driving `dbt compile`/`dbt parse` as a subprocess from an agent that also sources a `.env` defining that variable for unrelated reasons (this repo's own `DBT_PROJECT_DIR`, used by the ingestion recipe): a temp-copied project directory can get silently ignored in favor of the real one unless every dbt subprocess invocation passes `--project-dir`/`--profiles-dir` explicitly. Cost us one (harmless, verified) accidental write to the real project's `target/` cache before the fix.
